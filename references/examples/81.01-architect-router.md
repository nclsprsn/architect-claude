# Architect Router — Example Routing Interactions

**Engagement:** ACME Corp Customer Onboarding Modernisation — Phase 1
**Date:** 2025-10-28

---

## Interaction 1 — Engagement Start

**Input from Marcus Webb (Lead Architect):**

> "I've just been handed a programme brief from our CCO. She wants to modernise our customer onboarding process — currently 11 days average, she wants it under 3 days. We have no formal architecture work started. Where do I begin?"

---

### Phase / Intent Diagnosis

**ADM phase detected:** Preliminary / Phase A boundary — the engagement has a business mandate (brief from the CCO) but no architecture foundation: no agreed Architecture Principles, no tailored architecture framework, no Statement of Architecture Work. `[proven]`

**Intent detected:** New engagement start — specification track. The architect is creating architecture foundations, not reviewing existing artifacts.

**Horizon:** H2 — scaling an emerging digital capability (≤ 3-day onboarding cycle) rather than optimising an existing flow (H1) or seeding a disruptive business model (H3). The architecture must be scoped to H2 delivery, not over-engineered for H3.

---

### Routing Recommendation

**Step 1 — Run `/preliminary` first.**

**Why:** The programme brief gives you a business mandate, not an architecture foundation. Without running Preliminary first, you have no agreed Architecture Principles to govern the work, no tailored architecture framework aligned to ACME's governance model, and no organisational model for EA. Any Phase A deliverable produced without Preliminary foundations will either be challenged at Architecture Board ("which principles does this Statement of Architecture Work conform to?") or will need to be rebuilt once principles are agreed. Skipping Preliminary is a second-order risk: it does not block Phase A from starting, but it guarantees rework in Phase B when the first capability assessment hits an unanswered governance question. `[proven — TOGAF ADM: Preliminary is explicitly prerequisite to Phase A]`

**What it produces:** Architecture Principles, tailored Architecture Framework, Organisational Model for EA, and a Request for Architecture Work. These become the governance baseline for every subsequent phase.

**Input to bring:** The CCO's programme brief, any existing IT strategy or enterprise policies, and the names and roles of the key stakeholders (Sarah Chen CCO, David Okafor CISO, Tom Hayward Customer Ops Director).

**Reversibility:** Two-way door — the Preliminary outputs are living documents; principles can be updated as the engagement matures. However, delaying Preliminary increases the cost of revision in later phases.

---

**Step 2 — Run `/architecture-vision` immediately after.**

**Why:** Once Architecture Principles are agreed, Phase A can proceed. The Architecture Vision skill produces the Statement of Architecture Work — the formal scope agreement that bounds the engagement and secures the CCO's sponsorship. It also produces the Communications Plan and the first stakeholder map. Without a signed Statement of Architecture Work, Marcus Webb has no formal mandate and no defined scope, which means scope creep is not just possible — it is structurally inevitable. `[proven]`

**What it produces:** Statement of Architecture Work, Architecture Vision document, Communications Plan, approved stakeholder list with roles and engagement model.

**Input to bring:** Preliminary outputs (Architecture Principles, tailored framework), the CCO brief, and any known constraints (budget, timelines, regulatory).

**Prerequisite gap check:** Confirm that ACME has an existing Architecture Repository before Phase A begins. If no Architecture Repository exists, this is a Preliminary output gap — flag to the Architecture Board Chair and create a placeholder repository before Phase A deliverables are produced.

---

### Output Discipline

**Confidence:** `[proven]` — new engagement with no prior TOGAF artifacts and a sponsor brief in hand is an unambiguous Preliminary → Phase A pattern.

**Reversibility:** Two-way door — starting with `/preliminary` does not lock out any subsequent phase. If scope later narrows to a single domain, the Preliminary outputs remain valid and can be scoped down.

**Named owner + review trigger:** Marcus Webb (Lead Architect) owns routing validation. Re-assess routing if scope expands beyond onboarding (e.g., if CCO requests a broader CX architecture mandate before Phase A is complete).

**Broad Responsibility:** The onboarding process handles customer personal data (identity verification, consent, account creation). Preliminary must establish a Data Protection principle and a GDPR compliance principle before any Phase B or Phase C design begins. Failure to establish these principles in Preliminary means every downstream phase will be making implicit data handling decisions without agreed governance — a regulatory exposure that compounds at each phase.

## Interaction 2 — Mid-Engagement Pre-Board Validation

**Input from Marcus Webb (Lead Architect):**

> "We've completed our Phase C Application Architecture — we have an integration architecture document covering 7 integration points. We're preparing for Architecture Board submission next week. What should we run before submitting?"

---

### Phase / Intent Diagnosis

**ADM phase detected:** Phase C complete, moving to pre-submission validation before Architecture Board. `[proven]`

**Intent detected:** Validation track — the architect has an existing artifact and needs a structured quality gate pipeline before formal board submission. This is not a specification task; it is a validation pipeline.

**Horizon:** H2 — consistent with the engagement context; the integration architecture must be validated at H2 depth (reliability, governance, scalability) not just H1 (point-to-point connectivity).

---

### Routing Recommendation

Run the **4-gate validation pipeline** in order. These are sequential gates, not parallel alternatives. Each gate catches a different class of failure; running them out of order wastes effort.

| Gate | Skill | What it catches | Why this position in the sequence |
|------|-------|-----------------|----------------------------------|
| 1 | `/principles-check` | Integration patterns that violate agreed Architecture Principles (e.g., direct point-to-point coupling where a mediation principle requires an API Gateway) | Must run first — if the document violates principles, fixes will affect the artifact before the deeper review begins. Reviewing an artifact that violates principles wastes Gate 2 effort. |
| 2a | `/architecture-review` | Chief architect critique — quality attributes (reliability, security, scalability, operability), sensitivity points, tradeoff analysis, failure modes across all 7 integration points | Runs after principles are confirmed — so the review is not distracted by principle violations already flagged in Gate 1. |
| 2b | `/risk-radar` | Risk register across all 7 integration points — probability × impact scoring, RAID log, named owners and review triggers | Runs in parallel with Gate 2a — risk identification is independent of the architecture review and both inform the completeness check in Gate 3. |
| 3 | `/artifact-completeness` | Completeness against TOGAF Phase C Application Architecture template — missing sections, undocumented decisions, gaps in the artifact that would cause Architecture Board to return it | Runs after Gates 2a and 2b, because the review and risk outputs may identify missing sections; the completeness check then validates the artifact including any additions made after the review. |
| 4 | `/compliance-review` | Conformance to Architecture Contract, Architecture Principles, and enterprise standards — the formal board-submission gate check | Runs last — it validates the artifact in its final state, after all earlier gates have been resolved. Running compliance before completeness is a sequencing error: an incomplete artifact cannot be compliant. |

---

### Gate Detail

**Gate 1 — `/principles-check`**

**Input to bring:** The Phase C Integration Architecture document and the Architecture Principles produced in Preliminary (specifically any integration, security, and data principles). If no Architecture Principles document exists — this is a critical prerequisite gap. Flag immediately to the Architecture Board Chair: a principles check cannot be performed without agreed principles, and an Architecture Board submission without principles conformance evidence will be returned.

**What it produces:** Principles conformance scorecard, list of violations (if any) with remediation guidance, and a conformance attestation for each applicable principle.

**Expected finding in this engagement:** Check Principle coverage for event-driven vs point-to-point coupling decisions across all 7 integration points, GDPR data minimisation at each integration boundary, and API security controls (mTLS, OAuth 2.0, or equivalent).

---

**Gate 2a — `/architecture-review`**

**Input to bring:** Phase C Integration Architecture document (after any Gate 1 fixes are applied). Seven integration points should each have: integration pattern, protocol, error handling, retry policy, SLA/SLO, and security model documented.

**What it produces:** Quality attribute assessment across reliability, security, observability, and operability; sensitivity and tradeoff map; failure mode analysis for critical integration flows; fix list with severity, owner, and review trigger.

**Warning:** If any of the 7 integration points lacks an error handling policy or retry strategy, the architecture review will surface these as findings. Resolve them before Architecture Board submission — a board submission with undocumented error handling on integration points will not pass Phase G governance.

---

**Gate 2b — `/risk-radar`**

**Input to bring:** Phase C Integration Architecture document. Any existing risk register entries from Phase A or Phase B should also be included — the risk radar will check whether integration-layer risks are consistent with earlier phase risk postures.

**What it produces:** Risk heat map (probability × impact), RAID log (Risks, Assumptions, Issues, Dependencies), named owners and event-based review triggers per risk.

**Expected finding in this engagement:** Integration risks typically include: vendor API availability (SLA breach risk), KYC verification webhook reliability, point-to-point coupling creating cascading failure chains, and data in transit protection gaps at external integration boundaries.

---

**Gate 3 — `/artifact-completeness`**

**Input to bring:** Phase C Integration Architecture document, updated after Gates 2a and 2b findings are resolved.

**What it produces:** Completeness scorecard against the TOGAF Phase C Application Architecture template, list of missing or underpopulated sections, a completeness verdict (pass / pass with conditions / fail).

**Note:** Do not submit to `/artifact-completeness` before resolving Gate 2a and 2b findings — the completeness check is most valuable when run against the near-final artifact, not the draft. Running it on a draft that will be heavily revised inflates iteration cost.

---

**Gate 4 — `/compliance-review`**

**Input to bring:** Phase C Integration Architecture document in its final post-Gate-3 state, the Architecture Contract (if signed), Architecture Principles, and the board submission checklist (if ACME's Architecture Board has one).

**What it produces:** Compliance assessment scorecard, conformance attestation, list of non-conformances with remediation path and severity, and a board-submission recommendation (approve / approve with conditions / defer).

**This is the last gate before board submission.** If `/compliance-review` returns non-conformances, do not submit to the Architecture Board — resolve them first. A board submission with known non-conformances will either be returned or will result in a conditional approval with remediation obligations that delay Phase D.

---

### Output Discipline

**Confidence:** `[proven]` — Phase C complete, pre-board submission, integration architecture document in hand is an unambiguous 4-gate validation pipeline pattern.

**Reversibility:** Two-way door — running the validation pipeline before board submission does not constrain any Phase D decisions. The only irreversible step in this sequence is the Architecture Board submission itself: once submitted and approved, the Phase C baseline is locked and future changes require a Phase H change request.

**Named owner + review trigger:** Marcus Webb (Lead Architect) owns the gate sequence execution. Architecture Board Chair owns the compliance review gate outcome. Re-route to `/change-management` if the board returns the submission and scope changes are required before resubmission.

**Broad Responsibility:** The Phase C Integration Architecture covers 7 integration points, at least one of which (KYC webhook) processes customer personal data. The compliance review gate (Gate 4) must confirm that GDPR data processing obligations are met at each personal-data integration boundary — including confirmation of a signed DPA for any external data processor (e.g., KYC SaaS vendor). This is not a documentation formality; it is a legal pre-condition for Architecture Board approval. `[proven — GDPR Art. 28]`
